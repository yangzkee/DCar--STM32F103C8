# DcarON Arduino 客户端 —— 调试说明（给同事）

把 DcarON 小车的 **DFCom_v2 协议**移植到 **Arduino Uno / Nano (ATmega328P)**。
协议/坐标系/scale 跟 STM32 版 DFCom_v2 **完全一致**，只换了平台层（Serial / millis / 字节流解析）。

麻烦你接上真实硬件帮忙调一下 —— **代码两版都已编译通过，但还没硬件实测。**

## 2. 编译 / 上传

**Arduino IDE**：库版 `文件→示例→DFCom→DcarON_Square`；或直接开单文件 `.ino`。板子选 `Arduino Uno`（Nano 选 `Nano + ATmega328P`）。

**arduino-cli**：
```bash
arduino-cli compile --fqbn arduino:avr:uno --libraries . DFCom/examples/DcarON_Square/DcarON_Square.ino
arduino-cli upload  -p /dev/cu.xxxxx --fqbn arduino:avr:uno DFCom/examples/DcarON_Square/DcarON_Square.ino
```

## 3. 接线（Uno/Nano 5V ↔ 车 USART3 3.3V）

```
Arduino D1(TX) → [电平转换 5V→3.3V] → 车 USART3_RX (PB11)
Arduino D0(RX) ←──────── 直连 ─────── 车 USART3_TX (PB10)
Arduino GND   ←─────────────────────  车 GND
(调试) Arduino D3 → USB-TTL 的 RX，9600 看 SoftwareSerial 打印
```
- ⚠️ **D1→车RX 一定要电平转换/分压**（5V 直怼 3.3V 引脚有风险）。
- ⚠️ **上传程序时先拔 D0/D1**（跟 USB 抢串口），传完再接车。

## 4. ★ 前置条件：车固件要带 USART3 回传镜像 ★

里程计回传默认走 UART4/USART1（460800，Uno 读不了）。我已在车固件 `DF_Bytes_Send()` 加了
**USART3 @115200 回传镜像**（`UART3_SendData` 改成等发完防共享缓冲损坏）。
**车必须烧这版固件**，Arduino 才收得到 odom / 完成回传。

> 指令方向（Arduino→车）不用改：USART3 RX 天生就在解析 DFLink，@115200。
> 需要车固件 `.hex` 找我要。

## 5. 重点排查顺序（按怀疑度）

1. **收不到任何回传** → 八成是车没烧 USART3 镜像固件，或 D0 没接对/电平问题。
   先单独验证：发一次 `dcarSubscribeOdom(10)`，示波器/逻辑分析仪看车 PB10(USART3_TX) 有没有 115200 数据出来。
2. **能收到 odom 但 `waitDone()` 卡死** → 完成回传(A=0x6F)没到，或 `cmd&0x07` 槽位对不上。
   可临时在 `dfDispatch` 里把收到的 A/B 打到 SoftwareSerial 看。
3. **数值不对/乱码** → 校验和或字节序问题；对照 §6 的帧格式。先确认波特率两端都 115200。
4. **丢帧/偶发卡顿** → SoftwareSerial 打印太频繁屏蔽中断丢了硬件串口字节。把调试打印减少、移出 `waitDone`。
5. **里程计订阅 >50Hz 不稳** → USART3 每帧 ~4.4ms，订阅太快回传挤不过来。先用 10Hz。

## 6. 协议帧速查（调试对照）

**发送帧**：`[0xDF][0x01][0x97][A][B][LEN][payload..][0xFD][csum_lo][csum_hi]`
校验和 = `[0]` 累加到 `[0xFD]`（含），u16 小端。
- 运动类 A=`0x02`：Linear B=`0x64`(LEN13)、Rot B=`0x63`(LEN8)、Vel B=`0x62`(LEN12)、Arc B=`0x66`(LEN13)、LinearWithYaw B=`0x65`(LEN17)。payload 全是 s32 LE ×10000(SI)，运动类末尾 1B profile(0/1)。
- 订阅 Odom: A=`0x04` B=`0x80`；订阅 VelPos: A=`0x04` B=`0x81`；payload 都是 `[type(1=持续)] [freq码(1=10Hz,5=50Hz...)]`。

**回传帧**：`[0xDF][0x97][0x01][A][B][LEN][payload..][0xFD][csum]`（地址顺序反过来）
- `A=0x6C B=0x80`：DF_odom 51B（version 0x04）→ 解到 `odom`
- `A=0x6C B=0x81`：DF_vel_pose 35B（version 0x02）
- `A=0x6F B=cmd`：完成回传 `[Process(0xFF=完成) Notice(0=正常)]` → `waitDone` 等这个
- `A=0x8C B=0x40`：状态回传 12B（imu_calibrated 等）

scale：角度×10000、acc×400、gyro×1800、速度×5000、位置(s32)×1000。

> 权威参考：STM32 版 `客户端例程_DFCom_v2/DFCom_Example/HARDWARE/DFCom_Tx.c` 和 `DFCom_Rx.c`，
> 字段布局一模一样，Arduino 版就是照它移植的。
